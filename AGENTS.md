## 零、研究启动前置规则（必守 · 防模糊启动）

**核心原则：用户可能是小白，问题天然宽泛。AI 是专业研究员——拿到模糊主题后，必须先向用户逐项确认关键参数，全部拿到后才能开始跑 skill。禁止在信息不全时直接启动扫描，禁止替用户"默认假设"。**

### 启动前必须确认的清单

当用户提出一个行业/主题/赛道后，按以下顺序逐轮询问（一次 2-3 个问题，不要一次丢 10 个）。

**询问格式硬规则**：每次向用户提问时，**必须给出至少 3 个具体可选项**供用户选择，不能只丢开放式问题让用户自己想。用户是小白，可能不知道有哪些选项——你要用你的研究能力预判出最有价值的分支，让用户从菜单里挑。示例：

- ❌ 坏：「你想看哪个市场？」
- ✅ 好：「市场范围你倾向哪个？A. 美股（流动性好、标的全）B. A 股（你熟悉、但要注意板块代码陷阱）C. 港股+美股混合 D. 全球扫（含台日韩欧，海外股数据可能有 gap）」

每轮用 `ask_choice` 工具给出 3-6 个带简短说明的选项，让用户点选。

**第一轮：锁定边界**
1. **主题精确化** — 用户说的是大类（如"AI 算力"）还是子领域（"AI 数据中心液冷"）？大类下通常有 3+ 个供应链特征差异显著的子领域，必须拆开分批跑。让用户确认：本期聚焦哪个子领域？其余的是否后续再开？
2. **市场范围** — 美股 / A 股 / 港股 / 台股 / 日股 / 韩股 / 欧股 / 全球？不指定默认全球，但要明确告知用户。

**第二轮：数据环境**
3. **API Keys** — 所有 key 统一存放在 `/home/ubuntu/Serenity/.env`，启动研究前自动读取。当前各 skill 依赖的 key：
   - `EODHD_API_KEY`：Bottleneck Hunter 价格拉取 + ticker 双向验证的核心依赖。没有 → 降级到 yfinance，海外股数据常有 gap。
   - `EDGAR_IDENTITY`：hasthaomni 五个子 skill + ZadAnthony 拉 SEC filings 的依赖。格式为 `"姓名 email@example.com"`。没有 → SEC 一手源无法拉取，只能靠网页间接获取。
   - 启动前检查 `.env` 中有哪些 key，缺什么明确告诉用户，让用户决定补不补。
4. **数据截止日** — 用户希望以哪天的数据为基准？（默认当天，但允许用户指定。）

**第三轮：偏好与约束**
5. **市值偏好** — 有无市值下限/上限？（如"只看 Sub-$2B""排除微盘 OTC""只要大市值龙头"）
6. **流动性/准入限制** — 是否接受 OTC / 海外交易所 / 港股通不可投标的？不接受的要提前排除。
7. **已知排除项** — 用户心中已排除的标的/子行业？（如"不要中国军技""不要 SaaS""排除 TSLA"）

**第四轮：节奏确认**
8. **深度预期** — 走完整四阶段闭环，还是只跑到某个阶段？（默认全跑，但如果用户只是想快速摸底，可以只跑阶段①。）
9. **时间容忍度** — 全流程是重任务（广扫 20+ 标的、ETF audit、逐票深挖、量化复核），可能需要多轮对话。用户是否准备好？

### 规则

- **所有 API Key 统一从 `/home/ubuntu/Serenity/.env` 读取**。启动研究前先读该文件，确认有哪些 key 可用、缺哪些。
- **每次研究任务必须产出 `.md` 文档**，保存到 `reports/` 目录。文档含完整研究记录和结论：确认的参数清单、各阶段产出、价格数据、财务数据、判定依据、交叉验证分歧点、数据缺口。文件名格式：`reports/<主题>_研究报告_<日期>.md`。
- **最终排名必须含完整公司信息**，每家公司至少包含：公司全称、所属国家、所有交易所代码（主代码 + ADR/替代代码）、在哪个交易所可以买到、市值、市盈率 PE、营收规模、该主题相关业务的营收占比（%）、在产业链中的具体角色与市占率、当前价格与动量数据（Rng / 1m / 3m / Stage）、详细财务数据（如有 SEC/年报来源则提取三年对比）。
- **日股等非美股公司**如无法获取一手财务数据，必须标注 `[推断]` 并说明数据来源限制。美股公司优先从 SEC EDGAR（10-K/10-Q）提取一手数据。
- **用户没回答完清单之前，不启动任何 skill。**
- 禁止替用户"默认假设"——比如"你应该是想看美股吧""没提市值就默认不限"。
- 如果用户说"你帮我定"，就按最保守选项（全球 / 不限市值 / 全跑 / 当天数据 / 有 key 用 key 无 key 降级），但**必须把选了什么逐项告知并让用户确认**。
- 用户中途改变主意（如"算了只看美股吧"）→ 立即更新约束，后续步骤不再跨越。

二、四技能的详细差异与验证模式
🧭 阶段①：源头扫图（Bottleneck Hunter + muxuuu）
目的：面对一个模糊主题，快速摸出产业链瓶颈和候选方向。

Bottleneck Hunter：最像"作战底图"的工具，给定一个主题会自动建立产业链网状图，用双规则判定瓶颈节点（漏斗型与枢纽型），输出候选晋级榜与水位标尺，最后所有判断都会落到 tracking/forward_picks.csv 里，方便事后跟踪。

muxuuu：更像标准化的"执行监督引擎"。它的研究路径极其完整：热点拆解 → 产业链分层 → 瓶颈环节研判 → 优选候选与风险排查。同时会要求 AI 用"公告、财报、问询函、互动易、招投标"等交叉验证证据，堵住单一叙事风险。

交叉验证用法：先用 Bottleneck Hunter 生成产业链底图 + 瓶颈候选表，再用 muxuuu 的标准化框架对同一主题做一次平行推演，对比二者在瓶颈节点判断上的分歧点。分歧本身就是最佳分析入口。

🧠 阶段②：龙头深挖（ZadAnthony）
目的：当拿到一个具体标的，一票一票彻底拆穿其逻辑和风险。

这是四个技能中唯一完整自带 防自欺闭环 的分析工具，完整实现：给定题目 → 科普底册 → 信念显形与逻辑判据 → 客观估值三档区间 → 买卖五档定档 → 风险与"什么会证伪" → 独立复核者机制。取数时强制按一手源 / 管理层声称 / 推断 / 推测分级标注，避免幻觉侵蚀决策。适合与 muxuuu 的候选清单结合，逐一深挖。

交叉验证用法：将阶段①产出的候选标的，投入 ZadAnthony 逐票深挖，重点关注它的 "什么会证伪" 清单——一个逻辑到底靠不靠得住，看它最怕什么条件逆转。

📊 阶段③：信号量化与漏斗验证（hasthaomni）
目的：用数学化方式复核α可信度、估值合理性、趋势健康度。

它不是一个技能，而是5 个子 Skill 组成的工具箱：

serenity-alpha：把市场消息拆成 alpha 假设 → 需求 → 财报 → 小盘弹性 → 验证路径，强在可追溯的 1–4 季度验证链与可证伪点

bayesian-intrinsic-growth-valuation：用贝叶斯假设更新估算 3–5 年内生增长，并与市场隐含增长与 FOMO 比较

gf-dma-health-index：综合估值与趋势健康度评分

tam-adj-peg：基于 TAM 口径调整 PEG 做成长估值

buy-side-equity-research-memo：生成带来源引用的买方股权研究备忘录，含 Serenity 框架交叉核验

交叉验证用法：当 ZadAnthony 输出一个"强信念标的"后，马上用 hasthaomni 子技能进行多维度量化复核，特别关注 α 假设的验证链条是否有断裂、内生增长与市场定价是否存在不合理差距。如果量化信号与前面质性判断冲突，就要重新审视整个逻辑。

三、典型的交叉验证工作流（建议串行执行）
阶段	使用技能	产出什么	验证重点
① 撒网	serenity-bottleneck-hunter + muxuuu	产业瓶颈图 + 候选清单	两个工具的推演结论是否一致 → 分歧点意味着未被充分定价的信号
② 收网	ZadAnthony	逐票深挖 + 三档估值 + 证伪点清单	选票逻辑是否严丝合缝 → 证伪边界是否清晰
③ 量化	hasthaomni 子技能	α 验证链 + 贝叶斯增长 + 健康指数 + TAM调整PEG + 买方备忘录	α 假设是否自洽、估值是否偏离基本面、边际趋势与基本面是否背离
四、最重要的小贴士：四者不要"串行等待"，要"并行对账"
这四个技能在设计哲学上天然适合做 平行推演与交叉对账：

Bottleneck Hunter 自己就是一把"铁标尺"，每份报告中的候选晋级榜、水位标尺与判定史天然可做回溯验证。

muxuuu 的标准框架会提示 AI 去查验原始公告等一手源信息，对 Hunter 的输出做二次交叉校验。

ZadAnthony 是最严苛的逻辑与估值对账端——任何经不起它"防自欺复核"或"强制证伪点"检验的标的，都不应进入实质决策。

hasthaomni 的 α 验证链条和增长估算可以从量化端复核整个逻辑的基础假设：如果 alpha 假设在市场验证链条上断裂、或贝叶斯增长严重低于预期，那么前三个技能产出的结论都值得重新检验。

总结：把一个主题同时喂给 Bottleneck Hunter 与 muxuuu 做平行扫图 → 对产出的候选清单使用 ZadAnthony 逐票深挖 → 用 hasthaomni 量化复核逻辑假设与估值区间。哪个环节出现强烈冲突，哪里就是最值得你本人介入深度思考的地方。

## 六、项目 Memory 规则（必守）

本项目使用 `agent_memory/` 维护项目级上下文，在以下情况**必须先读取** `agent_memory/context.md`、`agent_memory/progress.md`、`agent_memory/bugs.md`：

- 多步骤任务、跨会话继续任务
- 策略开发、回测、交易系统维护、复杂 bug 排查
- 涉及运行时代码、构建、测试、类型、公开接口或关键 UI 的改动
- 用户说"继续""接着上次""按之前的""恢复进度"

### 文件结构

| 文件 | 内容 |
|------|------|
| `agent_memory/context.md` | 项目背景、目标、范围、架构、关键约束、外部依赖、稳定决策 |
| `agent_memory/progress.md` | 当前任务进度、已完成事项、下一步、阻塞点、待用户确认项 |
| `agent_memory/bugs.md` | 已发现问题、失败尝试、根因判断、风险、待验证点 |
| `agent_memory/archive/` | 过期或过长内容的归档 |

### 写入规则

出现以下情况必须更新对应 memory：阶段完成 / 方向变化 / 用户确认关键决策 / 发现或修复重要 bug / 失败尝试有后续价值 / 新增重要约束或风险 / 会话即将结束且后续需继续。

- 稳定背景、目标、架构、约束、决策 → `context.md`
- 当前状态、完成情况、下一步、阻塞点 → `progress.md`
- bug、失败尝试、风险、验证缺口 → `bugs.md`

### 内容质量

- 不写流水账，不记录一次性临时信息，不复制大段日志
- 每条记录必须对后续判断、执行或风险控制有价值
- 文件过长时先压缩为摘要，旧内容移至 `agent_memory/archive/`
- 跨项目复用的经验沉淀到全局 skill，不留在项目 memory

### 完成标准

长期项目或多轮任务完成前，确认 `context.md` / `progress.md` / `bugs.md` 是否需要更新。本次无需更新则简短说明"本次无项目 memory 更新"。
